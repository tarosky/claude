# 抽出サマリ スキーマ

ステップ4の各サブエージェント（1リポジトリ = 1エージェント）が返す構造化サマリの形式。
**read-only 調査**の結果を、以下の Markdown 構造で返すこと。ファイル内容の丸ごと転記は禁止。
関連ファイルは**パス**で示す（必要な原文は合議フェーズで都度読む）。

```markdown
## <owner/repo>

- **theme_handled**: このリポがテーマをどう扱っているかの1〜3文の要約
- **status**: implemented | partial | absent | custom
  - implemented: テーマの機能が一通り揃っている
  - partial: 一部のみ
  - absent: 該当なし
  - custom: 独自色が強く、そのまま比較しづらい
- **related_files**: テーマに関係するファイルパス（存在するもののみ）
  - path/to/file1  — 役割の一言
  - path/to/file2  — 役割の一言
- **key_choices**: テーマ上の主要な技術選択（箇条書き）
  - 例: リリースは release-drafter + tag トリガー
  - 例: PHP 8.1 / WP 6.x マトリクス
- **strengths**: このリポのやり方の強み（箇条書き）
- **weaknesses**: 弱み・逸脱・古い部分（箇条書き）
- **notes**: 補足（バージョン、依存、特殊事情など。任意）
```

## 収集の指針

- テーマに **直接関係するファイルのみ** を対象にする。無関係なコードは調べない。
- バージョン・ツールチェーン（言語/フレームワーク/CI ランナー等）はテーマに関係する範囲で記録。
- 「良い/悪い」の判断は根拠つきで簡潔に。断定しすぎない。
- 見つからなかった場合は `status: absent` とし、related_files を空にする。
- **ファイルを変更しない。** 読み取りのみ。

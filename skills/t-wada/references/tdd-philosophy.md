# t-wada TDD哲学リファレンス

## 概念の階層

テスト駆動開発と混同されやすい概念を正確に区別する。上位の概念ほど広く推奨され、下位ほど「お好み」の領域になる。

```
自動テスト（Automated Test）        ← 強く推奨
  └ 開発者テスト（Developer Testing）  ← 強く推奨
    └ テストファースト（Test-First）    ← 選択肢
      └ テスト駆動開発（TDD）          ← 選択肢
```

- **自動テスト**: テスティングフレームワークを使ってテストコードを書くこと。誰がいつ書いても構わない
- **開発者テスト**: 開発者自身が自動テストを書きながら開発すること。実装タイミングに近いほど効果が高い
- **テストファースト**: テストコードを実装より先に書くこと。TDDの必要条件だが十分条件ではない
- **テスト駆動開発（TDD）**: テストリスト + テストファースト + インクリメンタル設計 + リファクタリングを組み合わせたプログラミングワークフロー

> 「テスト駆動開発の効果」と言われているものの多くは、実際には自動テストや開発者テストの効果である。自動テストを書きましょう。開発者テストを行いましょう。この2つは本当におすすめです。それらに比べれば、テストファーストやテスト駆動開発は「お好み」です。— t-wada

## Kent Beckの5ステップワークフロー

TDDは「Red → Green → Refactor」の3ステップとして広まったが、これは意味の希薄化（Semantic Diffusion）の結果である。Kent Beck自身が定義する正確なワークフローは以下の5ステップ:

1. **テストリスト**: 新しい振る舞いにおいて期待される動作をリストアップする（振る舞いの分析）
2. **Red（テスト作成）**: リストから「ひとつだけ」選び、具体的で実行可能なテストコードに翻訳し、失敗を確認する
3. **Green（テスト成功）**: プロダクトコードを変更し、そのテスト（と既存の全テスト）を成功させる
4. **Refactor（リファクタリング）**: 必要に応じてリファクタリングを行い、実装の設計を改善する
5. **繰り返し**: テストリストが空になるまでステップ2に戻る

> より正確に表現するなら「リスト、レッド、グリーン、リファクタ」である。— t-wada

多くの人が書籍『テスト駆動開発』の中に出てくるステップ1（テストリスト）を見逃している。「TDDはいきなりコードを書き始める。いつ終わるのかも全然見通せない」という意見は、全くの見当外れである。

## インターフェース設計と実装設計の分離

TDDには2種類の設計判断が明示的に分離されている:

- **ステップ2（テスト作成）= インターフェース設計**: ある振る舞いはどのように呼び出されるべきかの設計。論理的・外部視点。テストを書く過程で自然にインターフェースが決まる
- **ステップ4（リファクタリング）= 実装設計**: システムはその振る舞いをどう実装するべきかの設計。物理的・内部視点。テストが通った安全な状態で内部構造を改善する

Kent Beckはこの2つを「論理設計と物理設計」と呼び、**決して混ぜてはならない**と述べている。TDDのワークフローは、この分離を自然に実現する仕組みになっている。

テストが書きにくいコードは、インターフェース設計に問題がある兆候である。

## Baby Steps

一度にひとつのことだけを変える。変更が小さいほど:
- 何が壊れたか特定しやすい
- git bisect が効果的に機能する
- レビュアーが変更を理解しやすい

テストリストの各項目を小さく保つことが、Baby Stepsの出発点となる。

## 各ステップでのよくある過ち

### ステップ1（テストリスト）の過ち

- **実装の設計判断を混ぜ込む**: 内部実装がどうあるべきか吟味する時間は後から確保できる。目の前のこと（振る舞いの分析）だけに集中する

### ステップ2（テスト作成）の過ち

- **テストリストの全項目を先にテストコードに翻訳してしまう**: 最初のテストを成功させる過程で設計をやり直したくなったら、投機的に書かれたテストは全部書き直しになる。ひとつずつ書くこと
- **アサーションのないテストを書く**: カバレッジのためだけのテストコードには価値がない。準備・実行・検証（アサーション）が揃った本物のテストをひとつだけ書く

### ステップ3（テスト成功）の過ち

- **期待値のコピペ**: テスト対象を実際に動かしたときの値をコピーして期待値にペーストすると、ダブルチェックにならずTDDの妥当性確認としての価値が台無しになる
- **テスト成功の過程にリファクタリングを混ぜ込む**: 「帽子を2つかぶる」問題。まず動かし、それから正しくする

### ステップ4（リファクタリング）の過ち

- **過剰なリファクタリング**: コードの整理整頓は気持ちがいいが、次のテストに向かうことを先延ばしにしてはいけない
- **早すぎる抽象化**: 重複はヒントであって、指令ではない

### 全般的な過ち

- **CIでテストが実行されない**: テストが存在してもCIで実行されなければ形骸化する
- **テスト実行が遅い**: 遅いテストは実行されなくなる。単体テストと統合テストを分離する

## power-assert: 失敗が語るアサーション

テストが失敗した時、失敗メッセージだけで原因がわかるべき。

```javascript
// Bad: 何が失敗したかわからない
assert.ok(result);

// Better: power-assert なら式の各部分の値が表示される
assert(response.body.users.length === 3);
// =>
// assert(response.body.users.length === 3)
//        |        |    |     |      |
//        |        |    |     2      false
//        |        |    [{...},{...}]
//        |        {users: [{...},{...}]}
//        {status: 200, body: {users: [...]}}
```

## 言語別推奨構成

### JavaScript / TypeScript

| カテゴリ | 推奨 | 理由 |
|---------|------|------|
| フレームワーク | Vitest（新規）, Jest（既存） | VitestはESM対応で高速。既存Jestは移行コストに見合わないことが多い |
| アサーション | power-assert または Vitest expect | power-assertは失敗時の情報量が圧倒的 |
| カバレッジ | v8（c8 / @vitest/coverage-v8） | IstanbulよりV8ネイティブカバレッジが高速 |
| モック | フレームワーク内蔵（vitest mock / jest mock） | 外部ライブラリ不要 |

### PHP

| カテゴリ | 推奨 | 理由 |
|---------|------|------|
| フレームワーク | PHPUnit | デファクトスタンダード。Pestはラッパーとして検討可 |
| カバレッジ | pcov | Xdebugより高速。CI環境では特に有効 |
| 静的解析 | PHPStan（Level 5以上） | テストと静的解析は補完関係 |
| モック | PHPUnit内蔵 / Mockery | 複雑なモックが必要な場合のみMockery |

### Python

| カテゴリ | 推奨 | 理由 |
|---------|------|------|
| フレームワーク | pytest | unittestより簡潔。fixtureの仕組みが強力 |
| カバレッジ | pytest-cov（coverage.py） | pytest統合が自然 |
| 型チェック | mypy / pyright | 動的型付け言語ではテスト+型チェックで安全性を確保 |
| モック | pytest-mock（unittest.mock wrapper） | pytestスタイルで記述可能 |

### Go

| カテゴリ | 推奨 | 理由 |
|---------|------|------|
| フレームワーク | go test（標準） | 外部フレームワーク不要。testifyは便利だが必須ではない |
| カバレッジ | go test -coverprofile | 標準ツールで十分 |
| リンター | golangci-lint | 複数リンターを統合実行 |

### Rust

| カテゴリ | 推奨 | 理由 |
|---------|------|------|
| フレームワーク | cargo test（標準） | 外部フレームワーク不要 |
| カバレッジ | cargo-tarpaulin | Rust向けカバレッジの定番 |
| リンター | clippy | 標準ツール |

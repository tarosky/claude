# デザインプロセス（兄弟プラグインとの差別化・機能からのモチーフ設計）

## なぜ調査が必要か

Tarosky は `taro-xxx` / 独自名のWordPressプラグインを多数運営している（`ts-plugins-org` 配下）。ロゴをそのまま使い回すと、WordPress.org の検索結果やプラグイン一覧で見分けがつかなくなる。ユーザーからも「他にもtaro-xxxというプラグインがいっぱいあるので、全部同じになったら困る」と明確な要望がある。**icon.svg は毎回そのプラグイン固有の機能を表すピクトグラムを新規に設計する。**

## 手順1: 実際の機能を確認する（プラグイン名で早合点しない）

プラグイン名やリポジトリ名から機能を推測しない。実例: `taro-lead-next` は名前だけ見ると「リード獲得（マーケティング）」に読めるが、実際は「投稿のページネーション用ブロック（次のページへ誘導する）」だった。README.md の説明文と、プラグイン本体のヘッダーコメント（`Description:`）を両方読んでから機能を要約すること。

## 手順2: 兄弟プラグインの資産を集めて見比べる

同じ組織内（例: `ts-plugins-org/*`）の他プラグインの `.wordpress-org/icon.svg` や `icon-256x256.*` を探して並べる:

```bash
for d in /path/to/ts-plugins-org/*/; do
  find "$d" -maxdepth 3 \( -iname "icon.svg" -o -iname "icon-256x256*" -o -iname "banner-*" \) 2>/dev/null
done
```

見つかったアイコンは Chrome DevTools MCP または（PNG/JPGなら）`Read` でそのまま目視確認する（SVGはImageMagickでなくChromeでレンダリングする。`references/rendering.md` 参照）。

## 手順3: 意匠のパターンを言語化する

過去の実例（2026年時点）:

| プラグイン | 意匠パターン |
|---|---|
| external-permalink | 円をブランドカラーで斜め2色分割 + 白の斜め矢印（外部リンク） |
| taro-taxonomy-blocks | 黒と青の正方形を少しずらして重ね、黒地に白のタグアイコン |
| taro-series | フラットな本のアイコン（青・黒） |
| taro-ad-fields | 青地に白の "AD" カードを2枚 |
| cookie-tasting | 写真（クッキー） |

**同じ「円をブランドカラーで斜め2色分割 + 白のピクトグラム」というテンプレート自体は複数プラグインで再利用してよい**（ブランドの統一感になる）。ただし中央のピクトグラムは必ず機能固有のものにする。ピクトグラムが他のプラグインと衝突しないことを確認してから確定する。

## 手順4: ピクトグラムを決める

機能を一言のメタファーに落とす。例:
- ページネーション「次へ」→ 二重シェブロン `»`
- 外部リンク → 斜め矢印
- タクソノミー/タグ → タグアイコン
- 広告枠 → カード/枠のアイコン

シンプルな幾何学図形（矢印・シェブロン・アイコン的ピクトグラム）に還元できないか検討する。複雑な絵柄よりも、256x256でも128x128でも視認できる太くシンプルな形を優先する。

## 手順5: banner はロゴを主役にしてよい

icon.svgとは逆に、banner は「Taroskyのこのプラグイン」であることを示す場なので、公式ロゴ（`horizontal-wh_rgb.svg` 等）をそのまま signature として使ってよい。構成の目安:
- 左: プラグイン名（大きく） + 一言タグライン（日本語）
- 右: icon.svg のモチーフを拡大したヒーログラフィック（背景色に合わせて配色を反転してもよい。ブランドガイドの「ライト/ダーク」展開と同じ考え方）
- 隅（右下推奨）: 公式ロゴの白版 or カラー版を小さく配置（署名的に）

## 手順6: ユーザーに方針を確認してから手を動かす

最低限、次の2点は `AskUserQuestion` で確認してから実装に入る（過去のセッションで実際にこの順で確認し、認識のズレを防げた）:
1. icon.svg は「公式ロゴそのまま」か「機能特化の独自ピクトグラム」か
2. banner に公式ロゴを使ってよいか、使う場合の役割（主役 or 署名）
